## 1. Архитектурная схема

### 1.1. Компоненты системы

- **Client**
  - Интерфейс для двух типов пользователей:
    - Менеджер: аутентификация, загрузка аудиозаписей, просмотр списка собственных записей с фильтрацией по датам.
    - Администратор: аутентификация, просмотр всех записей, управление режимом обработки, одобрение и повторный запуск транскрибаций.

- **API**
  - Функции:
    - аутентификация и авторизация (с помощью JWT токенов) с ролями `MANAGER` и `ADMIN`
    - приём файлов
    - сохранение метаданных записей в БД
    - загрузка файлов в Object Storage и сохранение их ключа в БД
    - создание задач транскрибации и постановка их в очередь
    - чтение-изменение режима обработки
    - одобрение задач и ручной повторный запуск (для администратора).

- **DB**
  - Хранит метаданные:
    - пользователи (логин, роль, хэш пароля)
    - клиенты
    - аудиозаписи (ID клиента, ID менеджера, дата загрузки, ссылка на файл в Object Storage, статусы, транскрипт)
    - задачи транскрибации (статус, режим, количество ретраев, ошибки, одобрение администратора)
    - режим обработки.
  - В таблицах хранится только ключ объекта в хранилище.

- **Object Storage**
  - Хранит исключительно бинарные аудиофайлы.
  - Ключ сохраняется в БД как ссылка на файл.

- **Queue (очередь задач - Kafka)**
  - Хранит сообщения о задачах транскрибации.
  - Сообщение содержит id задачи транскрибации.
  - Обеспечивает асинхронность и механизм ретраев (повторной доставки сообщений, отложенные очереди).

- **Worker (обработчик задач транскрибации)**
  - Подписан на очередь.
  - Для каждой задачи:
    - читает метаданные задачи и записи из БД
    - скачивает файл из Object Storage по ключу
    - вызывает внешний ИИ-сервис транскрибации
    - сохраняет текст транскрипта и обновляет статусы в БД
    - реализует стратегию Retry при ошибках.

### 1.2. Взаимодействие компонентов

1. **Client - API**
   - Запросы аутентификации.
   - Загрузка аудиозаписей.
   - Получение списка и информации записей.
   - Управление режимом обработки и задачами (для администратора).

2. **API - Object Storage**
   - Приём аудиофайла от клиента и его загрузка в объектное хранилище.
   - Получение ключа аудиозаписи для дальнейшего хранения в БД.

3. **API - DB**
   - Создание и обновление записей в таблицах.
   - Чтение метаданных для отображения в клиенте и для постановки задач.

4. **API - Queue**
   - Публикация сообщений о задачах транскрибации:
     - сразу при загрузке записи в режиме Авто;
     - при одобрении задач администратором в ручном режиме;
     - при ручном Retry администратором.

5. **Worker - Queue**
   - Получение задач транскрибации.
   - При ошибках — повторная публикация (либо в ту же, либо в отложенную очередь) до достижения лимита ретраев.

6. **Worker - Object Storage**
   - Загрузка аудиофайла по ключу для передачи ИИ.

7. **Worker - DB**
   - Обновление статусов задач и записей.
   - Сохранение результата транскрипции.

## 2. Модель данных

### 2.1. Таблица `users`

- **Поля:**
  - `id`
  - `email`
  - `password_hash`
  - `role`
  - `created_at`
  - `updated_at`

- **Ограничения и связи:**
  - PK: `id`.
  - Уникальный индекс по `email`.
  - Бизнес-логика по ролям реализуется на уровне API.

### 2.2. Таблица `clients`

- **Поля:**
  - `id`
  - `name`
  - `created_at`

- **Ограничения:**
  - PK: `id`.

### 2.3. Таблица `recordings`

- **Поля:**
  - `id`
  - `client_id`
  - `manager_id` FK - `users` (ожидается роль `MANAGER`).
  - `uploaded_at`
  - `file_key`
  - `status` ENUM('UPLOADED', 'PROCESSING', 'PROCESSED', 'FAILED'), NOT NULL, DEFAULT 'UPLOADED'. 
  - `transcript`
  - `duration_seconds`
  - `created_at`
  - `updated_at`

- **Ограничения и связи:**
  - PK: `id`.
  - FK: `client_id` - `clients(id)`.
  - FK: `manager_id` - `users(id)`.

### 2.4. Таблица `transcription_tasks`

- **Поля:**
  - `id`
  - `recording_id`
  - `mode_at_creation` ENUM('AUTO', 'MANUAL'), NOT NULL.
  - `status` ENUM('PENDING', 'QUEUED', 'IN_PROGRESS', 'COMPLETED', 'FAILED', 'CANCELLED'), NOT NULL, DEFAULT 'PENDING'.
  - `retry_count`
  - `max_retries`
  - `text_error`
  - `approved_by_admin_id` FK - `users`.
  - `approved_at`
  - `queued_at`
  - `started_at`
  - `completed_at`
  - `created_at`
  - `updated_at`

- **Ограничения и связи:**
  - PK: `id`.
  - FK: `recording_id` - `recordings(id)`.
  - FK: `approved_by_admin_id` - `users(id)`.


### 2.5. Таблица `processing_settings`

- **Поля:**
  - `id`
  - `mode` ENUM('AUTO', 'MANUAL')
  - `updated_by_admin_id` FK - `users`.
  - `updated_at`

- **Ограничения:**
  - PK: `id`
  - FK: `updated_by_admin_id` - `users(id)`.

## 3. Контракты API

### 3.1 `POST /auth/login`

- Авторизация: не требуется
- Запрос (JSON):
```json
{ "email": "manager@example.com", "password": "secret-password" }
```

- Ответ (200):

```json
{
  "access_token": "...",
  "refresh_token": "...",
  "user": { "id": "...", "role": "MANAGER" }
}
```

- Ошибки: `400`, `401`, `500`.

### 3.2. `POST /recordings`

- Авторизация: требуется (роль: Менеджер).
- Запрос: `multipart/form-data`
  - `file`
  - `client_id`
  - `uploaded_at`

- Ответ (201):
```json
{
  "id": "...",
  "client_id": "...",
  "manager_id": "...",
  "uploaded_at": "...",
  "status": "UPLOADED",
  "file_key": "...",
  "transcription_task": { "id": "...", "status": "QUEUED" }
}
```

- **Ошибки:** `400`, `401`, `403`, `500`.

### 3.3. `GET /recordings`

- Авторизация: требуется (роль: Менеджер / Админ).
- Query: `from` (ISO), `to` (ISO), `client_id` (UUID), `manager_id` (UUID).

- Ответ (200):

```json
[
  { "id": "...", "uploaded_at": "...", "status": "PROCESSED" }
]
```

- Ошибки: `400`, `401`, `403`, `500`.

### 3.4. `GET /recordings/{id}`

- Авторизация: требуется (роль: Менеджер /Админ; менеджер видит только свои записи).

- Ответ (200):
```json
{
  "id": "-...",
  "client_id": "...",
  "manager_id": "...",
  "uploaded_at": "...",
  "status": "PROCESSED",
  "file_key": "...",
  "transcript": "..."
}
```

- Ошибки: `401`, `403`, `404`, `500`.

### 3.5. Режимы транскрибации (Админ)

#### `GET /admin/transcriptionmode`

- Авторизация: требуется (роль: Админ).

- Ответ (200):

```json
{ "mode": "AUTO|MANUAL", "updated_at": "..." }
```

- Ошибки: `401`, `403`, `500`.

#### `PATCH /admin/transcription-mode`

- Авторизация: требуется (роль: Админ).
- Запрос (JSON):

```json
{ "mode": "AUTO|MANUAL" }
```

- Ответ (200):

```json
{ "mode": "MANUAL", "updated_at": "2026-03-02T15:45:00Z" }
```

- Ошибки: `400`, `401`, `403`, `500`.

### 3.6. Управление задачами транскрибации (Админ)

#### `GET /admin/transcriptiontasks`

- Авторизация: требуется (роль: Админ).
- Query: `status` (`PENDING`), `from`, `to`.

- Ответ (200):

```json
[
  { "id": "...", "recording_id": "...", "status": "PENDING", "retry_count": 0 }
]
```

- Ошибки: `401`, `403`, `500`.

#### `POST /admin/transcription-tasks/{id}/approve`

- Авторизация: требуется (роль: Админ).
- Запрос: `{}`

- Ответ (200):

```json
{ "id": "...", "status": "QUEUED", "approved_at": "..." }
```

- Ошибки: `400`, `401`, `403`, `404`, `500`.

#### `POST /admin/transcription-tasks/{id}/retry`

- Авторизация: требуется (роль: Админ).

- Ответ (200):

```json
{ "id": "...", "status": "QUEUED", "retry_count": 1 }
```

- Ошибки: `400`, `401`, `403`, `404`, `500`.


## 4. Описание ключевых процессов

### 4.1. Загрузка записи

**Цель:** от клика менеджера до сохранения файла в Object Storage и метаданных в БД.

1. Менеджер в Client выбирает аудиофайл и клиента, нажимает кно ку «Загрузить».
2. Client формирует запрос `POST /recordings` с полями `file`, `client_id` и заголовком `Authorization`.
3. API валидирует токен и проверяет роль `MANAGER`, проверяет существование клиента по `client_id`.
4. API отправляет файл в хранилище:
   - генерирует уникальный id файла;
   - загружает файл и получает подтверждение успешной операции.
5. API создаёт запись в БД в таблице `recordings`:
   - сохраняет `client_id`, `manager_id` (из токена), `uploaded_at`, `file_key`, начальный `status = 'UPLOADED'`.
6. API считывает `processing_settings.mode`:
   - если `AUTO`:
     - создаёт `transcription_tasks` с `status = 'QUEUED'`, `mode_at_creation = 'AUTO'`;
     - публикует сообщение в очередь.
   - если `MANUAL`:
     - создаёт `transcription_tasks` с `status = 'PENDING'`, `mode_at_creation = 'MANUAL'` (без постановки в очередь).
7. API возвращает клиенту ответ 201 с данными записи и созданной задачи.


### 4.2. Транскрибация (Авто-режим)


1. При загрузке записи в авто-режиме API создаёт задачу в `transcription_tasks` со статусом `QUEUED` и публикует сообщение в очередь.
2. Worker получает сообщение из очереди и:
   - обновляет задачу в БД: `status = 'IN_PROGRESS'`, `started_at = now()`.
3. Worker извлекает из БД запись `recordings` по `recording_id`:
   - получает id файла в хранилище и другие необходимые метаданные.
4. Worker скачивает аудиофайл из Object Storage по id.
5. Worker отправляет файл во внешний AI-сервис и ждёт результата.
6. При успешной транскрибации:
   - записывает текст в `recordings.transcript`
   - устанавливает `recordings.status = 'PROCESSED'`
   - устанавливает `transcription_tasks.status = 'COMPLETED'`, `completed_at = now()`.
7. При ошибке:
   - увеличивает `retry_count`
   - записывает текст ошибки в `text_error`
   - в зависимости от `retry_count` либо повторяет задачу, либо помечает задачу как `FAILED`.



### 4.3. Транскрибация (Ручной режим)

1. Администратор открывает UI со списком задач.
2. Client отправляет запрос `GET /admin/transcription-tasks?status=PENDING`.
3. API возвращает список ожидающих задач.
4. Администратор выбирает конкретную задачу и нажимает «Одобрить».
5. Client вызывает `POST /admin/transcription-tasks/{id}/approve`.
6. API:
   - валидирует роль `ADMIN`
   - проверяет, что задача в статусе `PENDING`
   - устанавливает:
     - `status = 'QUEUED'`
     - `approved_by_admin_id` и `approved_at`
   - публикует сообщение о задаче в очередь.
7. Далее Worker обрабатывает задачу по сценарию авто-режима.



### 4.4. Обработка ошибок и механизм Retry


#### Автоматический Retry

1. Worker читает сообщение из очереди и начинает обрабатывать задачу.
2. Если возникает ошибка:
   - Worker увеличивает `retry_count` для соответствующей записи в `transcription_tasks`
   - записывает текст ошибки в `text_error`.
3. Если `retry_count < max_retries`:
   - Worker повторно ставит задачу в очередь
   - статус задачи устанавливается как `QUEUED`.
4. Если `retry_count >= max_retries`:
   - Worker устанавливает:
     - `transcription_tasks.status = 'FAILED'`
     - `recordings.status = 'FAILED'`
   - сообщение не возвращается в основную очередь

#### Ручной Retry

1. Администратор в UI открывает список задач со статусом `FAILED` через `GET /admin/transcription-tasks?status=FAILED`.
2. Для выбранной задачи он нажимает «Повторный запуск».
3. Client отправляет запрос `POST /admin/transcription-tasks/{id}/retry`
4. API:
   - валидирует роль `ADMIN`
   - проверяет статус задачи (`FAILED`)
   - обновляет:
     - `status = 'QUEUED'`;
     - корректирует `retry_count` и очищает `text_error`.
   - публикует задачу в очередь.
5. Worker получает задачу и вновь обрабатывает её по стандартному сценарию.
